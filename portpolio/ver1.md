# 핵심 역량 정의
Hips-Tour 프로젝트에서 여타 기술 블로그처럼 깊이 있게 다룰 수 있는 핵심을 분류
`자랑할 부분`을 명확하게 잡아야 한다

+ 문제 정의: API 호출 수 제한, 응답 지연 등 운영 환경의 리스크를 예측
+ 전략적 설계(해결 방향이 되겠지): 초기 적재`Load`와 증분 동기화`Sync`를 분리, 페이지 단위 `Bulk 처리`로 API 호출을 줄였다.
  + 상황에 맞는 최적의 전략을 설계한 부분이 될것
+ 장애 방지 설계(예외 처리같은)
  + `중단/재개` 로직으로 배치 작업의 안정성을 확보했어
  + 2단계 데이터 보강으로 데이터 무결성을 확보했어(결국 101번째 호출하는 상황이 되어버렸어)

## 포트폴리오 재구조화 및 기존 자료 재활용 (살릴 수 있는것들은 살리자 - 기회비용이 커)
### 포트폴리오의 흐름 및 구조 정리
+ 프로젝트 개요
  + 프로젝트 소개, 기술 스택, Hips-Tour란 뭐하는애야? 및 그밖에 TourAPI같이 처음 등장하는 인물을 설명해야지
+ 시스템 아키텍쳐
  + 모듈형 모놀리스 였던거 같아. (상세 검색 및 프로젝트 내용과 비교 해보자)
    + 각 모듈은 책임 중심으로 구성되어있어
+ `핵심` 기술 문제 및 해결 과정 (Deep하게! 근데 그걸 어떻게? 기술 문서 참고해보자)
  + `이 부분이 집중해서 작성해야 할 메인 스토리`!!
  + 문제정의 > 나의 생각? > 해결 과정? > 결과 (최적화 되었는가? 안정성이 확보 되었는가? 그밖에 범한 오류는 없는가? 같은걸 생각해보자)
+ 기타 주요 설계
  + 우선 중점은 sync로 가되, common과 imageUpload 같은 설명도 붙여줘야지
    + `Key`: N + 1 방지, 다른 api로 교체 가능하게 (유연한 프로그램) - 기본에 충실한 내용이 될거야
